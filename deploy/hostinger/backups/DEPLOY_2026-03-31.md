## Procedimento recomendado

### 1. Atualizar o código no servidor

```bash
cd /opt/ABASE/repo
git fetch origin
git pull origin abaseprod
```

Resultado esperado:

- fast-forward a partir de `2677aeb`
- chegada do commit deste pacote

### 2. Rebuild das imagens

Como houve alteração simultânea em backend e frontend, fazer rebuild completo de ambos.

```bash
docker compose -p abase --env-file /opt/ABASE/env/.env.production \
  -f deploy/hostinger/docker-compose.prod.yml \
  build --no-cache backend frontend
```

Se a sessão remota estiver instável:

```bash
nohup docker compose -p abase --env-file /opt/ABASE/env/.env.production \
  -f deploy/hostinger/docker-compose.prod.yml \
  build --no-cache backend frontend \
  > /tmp/abase_build_2026-04-01.log 2>&1 &
```

### 3. Recriar backend, celery e frontend

Este pacote altera código compartilhado do backend. Não é seguro deixar o worker Celery rodando com código antigo.

```bash
docker stop abase-backend-prod abase-celery-prod abase-frontend-prod
docker rm abase-backend-prod abase-celery-prod abase-frontend-prod

docker compose -p abase --env-file /opt/ABASE/env/.env.production \
  -f deploy/hostinger/docker-compose.prod.yml \
  up -d --no-deps --force-recreate backend

docker compose -p abase --env-file /opt/ABASE/env/.env.production \
  -f deploy/hostinger/docker-compose.prod.yml \
  up -d --no-deps --force-recreate celery

docker compose -p abase --env-file /opt/ABASE/env/.env.production \
  -f deploy/hostinger/docker-compose.prod.yml \
  up -d --no-deps --force-recreate frontend
```

### 4. Validar saúde dos containers

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
```

Esperado:

- `abase-backend-prod` em `healthy`
- `abase-celery-prod` em execução estável
- `abase-frontend-prod` em execução estável
- `abase-nginx-prod`, `abase-redis-prod` e `abase-mysql-prod` preservados

## Checklist funcional pós-deploy

### 1. Login web

- abrir `https://abasepiaui.cloud/login`
- autenticar com usuário administrativo
- confirmar navegação sem erro de bundle

### 2. Detalhe do associado

- abrir um associado existente
- confirmar badge de origem `Web` ou `Mobile`
- habilitar modo admin editor
- confirmar que não ocorre loop/render infinito

### 3. Tesouraria — Novos Contratos

- abrir `/tesouraria`
- validar busca, filtros avançados e KPIs
- confirmar pendentes reais na seção `Aguardando efetivação PIX`
- confirmar que contratos históricos efetivados aparecem conforme o período filtrado
- testar exportação e conferir que o relatório sai com as mesmas colunas da seção
- validar botões de visualizar e substituir anexos em contratos efetivados

### 4. Tesouraria — Pagamentos

- abrir `/tesouraria/pagamentos`
- confirmar que admin/tesoureiro usam a rota nova
- validar filtros avançados e organização da tabela

### 5. Tesouraria — Renovações

- abrir `/tesouraria/refinanciamentos`
- confirmar a seção operacional de renovações aprovadas para tesouraria
- validar anexação/efetivação no layout novo

### 6. Comissões

- abrir um contrato pendente e um efetivado na tesouraria
- validar que o valor da comissão do agente corresponde ao percentual aplicado sobre a base correta

### 7. Mobile API

- validar que `GET https://abasepiaui.cloud/api/v1/auth/login/` responde `405` em `GET`
- validar que `GET https://abasepiaui.cloud/api/v1/app/me/` sem token responde `401`
- se houver conta de teste controlada, validar no app mobile:
  - login
  - leitura de bootstrap
  - envio de cadastro básico

## Problemas já conhecidos e cuidados

### 1. Sempre usar `-p abase`

Evita subir serviços em projeto compose diferente do ambiente principal.

### 2. Sempre usar `--env-file /opt/ABASE/env/.env.production`

Evita containers sem variáveis obrigatórias.

### 3. Não validar login da API com `GET`

`/api/v1/auth/login/` é `POST`.

- `405` em `GET` é esperado
- isso não indica falha da rota

### 4. O app mobile não faz parte do deploy da Hostinger

Neste pacote, o que entra em produção para o mobile é:

- o alinhamento do backend `api/v1`
- a documentação do fluxo do `abase_mobile_new`

Não existe build Expo/EAS para publicar a partir deste roteiro de servidor.

### 5. Não incluir artefatos locais no git do servidor

Backups locais, dumps e relatórios de conciliação não fazem parte deste deploy e não precisam estar no servidor para a aplicação funcionar.
